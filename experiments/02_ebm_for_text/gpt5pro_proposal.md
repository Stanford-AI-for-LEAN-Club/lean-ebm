Here’s the version I would actually build.

For text, and especially for code and Lean4, I would not start with a pure EBM that tries to sample raw token strings by MCMC. I would use a strong proposal model (q_\phi) and learn a residual energy (E_\theta) over full candidates or full trajectories:
[
p_\theta(\tau \mid x)\propto q_\phi(\tau \mid x)\exp(-E_\theta(x,\tau)).
]
That matches what has worked best in text EBMs: residual sequence-level scoring on top of pretrained LMs, not token-local normalization. It also lines up with code EBM work that treats compilability as a global constraint, and with newer EBM-style language-model training that optimizes sequence-level behavior from model rollouts rather than only next-token loss. ([arXiv][1])

For reasoning, I’d move one step further and define the EBM over *trajectories* of tool-interacting states, not just final text. Recent energy-based reasoning work explicitly uses optimization over energy landscapes with adaptive test-time compute, and strong Lean systems now get gains from verifier-in-the-loop feedback, proof-progress prediction, and critic-guided reranking of intermediate states. That makes the EBM most valuable as a search critic or value model. ([arXiv][2])

**1. Model the right object**

Let (x) be the problem plus retrieved context. Let (\tau=(s_0,a_0,\dots,s_T)) be a trajectory, where (s_t) is the current code/proof state and (a_t) is an edit, tactic, or subgoal decomposition. Then use
[
E_\theta(x,\tau)=\sum_t e_\theta(s_t,a_t,r_t,d_t)+\lambda_v V_\theta(s_{t+1})+\lambda_g e^{global}*\theta(x,\tau).
]
Here (r_t) is retrieved repo/premise context, (d_t) is tool feedback, (e*\theta) is a local action energy, and (V_\theta) is a cost-to-go head. For Lean, (V_\theta) can approximate remaining proof steps; for code, expected remaining failing tests or remaining repair effort. LeanProgress is a direct precedent for the proof-distance piece. ([arXiv][3])

**2. Use structure, not only text**

For code, the state should include the task text, target file or span, AST or CST features, symbol or call-graph features, retrieved dependencies, compiler errors, static-analysis warnings, and test outcomes. RepoExec and CoCoGen both show that repository context and iterative compiler or debugging feedback matter a lot in repo-level generation. ([arXiv][4])

For Lean, I would encode the theorem statement, current goals, local hypotheses, tactic history, imported environment, and retrieved premises. I would not stop at pretty-printed goal text: Lean exposes `Expr` as the AST of Lean terms, `TacticM` as the tactic-state monad, and `MetaM` as the monad that gives expressions meaning through the environment and metavariable context. That means the energy model can consume elaborated proof states and metavariables, not just surface tokens. LeanDojo is the natural scaffold for collecting these states and search traces. ([Lean Community][5])

**3. Separate hard constraints from soft energy**

I would make parser, typechecker, or kernel failures hard masks whenever possible, rather than asking the EBM to learn them softly. In code, invalid syntax, type errors, or uncompilable patches should usually be rejected outright or assigned effectively infinite energy. In Lean, tactics that fail elaboration or produce invalid states should be pruned immediately. The soft part of the energy should focus on ranking *valid* options: which valid patch is most likely to pass tests, and which valid Lean step most reduces the distance to a finished proof. This follows the code-EBM framing of global compilability constraints and the Lean evidence that stepwise verifier feedback and state reranking are what improve search. ([arXiv][6])

A good step-energy decomposition is:
[
e_\theta=e_{\text{compat}}+\alpha e_{\text{tool}}+\beta e_{\text{progress}}+\gamma e_{\text{retrieval}}+\delta e_{\text{plan}}.
]
For code, (e_{\text{tool}}) uses compiler, tests, and static analysis, (e_{\text{progress}}) measures reduction in diagnostics or predicted failing tests, and (e_{\text{retrieval}}) scores API and dependency consistency. For Lean, (e_{\text{tool}}) uses verifier outcomes, (e_{\text{progress}}) predicts proof distance, (e_{\text{retrieval}}) scores premise usefulness, and (e_{\text{plan}}) checks consistency with a higher-level subgoal plan. Subgoal decomposition is especially worth modeling as a latent variable in Lean, because recent high-performing provers explicitly use it. ([arXiv][4])

**4. Train the EBM mostly from hard negatives generated by search**

I would not try to do exact EBM likelihood training over text. For a first serious version, I’d train with proposal-conditioned contrastive or listwise ranking losses over candidate sets sampled from the current proposer and search stack:
[
L=-\log \frac{\exp(-E_\theta(x,\tau^+))}{\exp(-E_\theta(x,\tau^+))+\sum_j \exp(-E_\theta(x,\tau^-_j))}.
]
The positives are successful proofs, passing patches, or strong prefixes that lead to success. The hard negatives are the valuable part: compile-but-fail code, patches with wrong repo API usage, Lean tactics that elaborate but dead-end, premise-retrieval distractors, semantically wrong formalizations, and backtracked nodes from best-first search. Residual text EBMs used contrastive training, and modern EBFT also supports the broader sequence-level view that rollout behavior matters more than token-wise teacher forcing alone. ([arXiv][1])

For data, I would combine human corpora with synthetic and self-supervised formal data. On the Lean side, PACT extracts extra training signal from kernel-level proof terms, DeepSeek-Prover shows the value of large synthetic Lean 4 proof corpora, and LeanNavigator massively expands the proof-data scale by generating new theorems and proofs from Mathlib4. ([arXiv][7])

**5. Inference should be search, not plain decoding**

At test time, run best-first or A*-like search over states. The proposer (q_\phi) generates the top (K) candidate edits or tactics, the tool executes or typechecks them, the EBM scores the successor states, and the value head estimates remaining work. Expand the frontier by low cumulative energy. For code, search over edits or patches rather than regenerating whole files. For Lean, search over tactic steps or subgoal decompositions, re-retrieving premises at each state. This is where the EBM pays for itself: more compute means deeper or wider search on hard instances, which is the same adaptive-compute pattern that energy-based reasoning work emphasizes. ([arXiv][2])

```text
frontier = {s0}
while budget remains:
    s = pop_best(frontier)
    R = retrieve_context(s)
    A = q_phi.sample_k(s, R)
    for a in A:
        s', diag = tool_execute(s, a)
        if invalid(s', diag):
            continue
        score = e_theta(s, a, R, diag) + λ * V_theta(s')
        push(frontier, s', cumulative_score + score)
    if solved(s):
        return extract_solution(s)
```

This exact loop is especially natural in Lean, because theorem-proving systems already benefit from progress predictors, verifier-in-the-loop local feedback, and external critics that rerank intermediate states. ([arXiv][8])

**6. Make context policy task-aware**

I would not use the same retrieval policy for ordinary code repos and repo-scale Lean verification. RepoExec finds that full dependency context often helps repository code generation, while VeriSoftBench finds that for repository-centric Lean verification, curated dependency-closure context beats dumping the whole repo, though the task remains hard. So the EBM should score candidates relative to a task-aware context controller, not a single fixed retriever. ([arXiv][9])

**7. For end-to-end verified Lean code, split the problem into three energies**

If the target is not just theorem proving but “write correct Lean code and prove it,” I would not use one monolithic energy. I’d factor it into
[
E_{\text{spec}}(x,\sigma),\quad
E_{\text{impl}}(x,\sigma,p),\quad
E_{\text{proof}}(x,\sigma,p,\pi).
]
That matches the real task structure much better. CLEVER explicitly separates specification generation from implementation synthesis and proof, and current models still struggle to get full end-to-end verification on that benchmark. On the proof side, subgoal decomposition should be a first-class latent plan variable, as in DeepSeek-Prover-V2. ([arXiv][10])

**8. What I would prototype first**

Version 1 would be a strong coder or prover proposal LM, a smaller bidirectional cross-encoder energy model, tool adapters for compiler and tests or LeanDojo and Lean 4, a retrieval module, and best-first search. I would train three heads on the energy model: pairwise or listwise ranking, a value head for remaining work, and a feedback head that predicts compiler or verifier error type. For code, a RankEF-style auxiliary target is useful because it teaches the model *why* code is wrong, not just whether; for Lean, the analogous auxiliary target is proof-progress. ([arXiv][11])

My strongest recommendation is this: for code and Lean4, treat the EBM as the **critic over structured search states**. That is much more plausible than trying to replace the base LM with a pure token-level EBM. I’m more confident in this architecture family than in any claim that pure discrete EBM sampling will beat a strong proposer-plus-critic stack. The real design choices are mostly where to put the boundary between hard pruning, learned value estimation, and expensive tool calls.
